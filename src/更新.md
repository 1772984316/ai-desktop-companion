## 1. 项目背景

### 1.1 产品定位

AI 桌面互动伴侣，常驻桌面的虚拟角色，能看、能听、能说、能主动互动、能帮用户做桌面操作。

**核心差异化：**

| 维度 | 传统 ChatBot | 本产品 |
|------|-------------|--------|
| 形态 | 聊天框 | 桌面虚拟角色 |
| 主动性 | 被动响应 | 主动触发问候/提醒 |
| 记忆 | 无/短期 | 长期记忆 + 关系养成 |
| 能力 | 纯对话 | 对话 + 截图理解 + 桌面操作 |

### 1.2 迁移动机

原方案依赖 Python `nanobot` 进程作为 AI Agent 后端，通过 WebSocket 与 Electron 通信：

```
Renderer → IPC → Main Process → WebSocket → Python nanobot Gateway (子进程)
```

**存在的问题：**
- 用户必须安装 Python 环境及 conda 虚拟环境，门槛高
- 启动链路长，冷启动需等待 Python 子进程就绪（最长 30s）
- 无法用 `electron-builder` 直接打包 Python 运行时
- 两套语言（Python + TypeScript）维护成本高

**决策：** 将 Agent 核心逻辑全面迁移至 TypeScript，直接运行在 Electron Main Process 内，彻底消除 Python 依赖。

## 3. 当前架构总览

### 3.1 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 客户端框架 | Electron 31 | 桌面应用容器 |
| 开发语言 | TypeScript 5.4 | 全栈统一语言，无 Python |
| 进程通信 | Electron IPC | Main ↔ Renderer 双向通信 |
| LLM 调用 | openai npm SDK | 兼容所有 OpenAI-compatible API |
| 会话存储 | JSONL 文件 | 本地持久化，轻量无依赖 |
| 长期记忆 | MEMORY.md | Markdown 文件，Agent 可读写 |
| 打包 | electron-builder | 单文件安装包，无需额外运行时 |

### 3.2 进程模型

```
Electron App
├── Main Process (Node.js)        ← Agent 全部逻辑在此运行
│   ├── AgentLoop
│   ├── LLMService
│   ├── ToolRegistry
│   ├── SessionManager
│   ├── HeartbeatService
│   └── IPC Handler
└── Renderer Process (Chromium)   ← 纯 UI，不含业务逻辑
    ├── chat.html / chat.ts
    └── preload.ts (contextBridge)
```

## 5. 核心模块说明

### 5.1 `agent/loop.ts` — AgentLoop

Agent 核心处理引擎，实现"思考 → 工具 → 回复"循环。

**处理流程：**

```
InboundMessage (来自 IPC Handler)
    ↓
SessionManager.getOrCreate(sessionKey)       // 恢复历史会话
    ↓
ContextBuilder.buildMessages(history + msg)  // 组装完整上下文
    ↓
┌─────────────── while iteration < maxIterations ────────────────┐
│   LLMService.chatWithRetry(messages, tools)                    │
│       ↓                           ↓                           │
│   has tool_calls               no tool_calls                   │
│       ↓                           ↓                           │
│   ToolRegistry.execute()       最终回复 → break               │
│   → 追加结果到 messages                                        │
└────────────────────────────────────────────────────────────────┘
    ↓
SessionManager.save(session)                 // 持久化本轮对话
    ↓
OutboundMessage → IPC → Renderer
```

**支持的斜杠命令：**
- `/new` — 清空当前会话，开始新对话
- `/help` — 显示可用命令

**进度推送：** 工具调用过程中通过 `onProgress` 回调向渲染进程发送 `partial=true` 的中间消息，实现"正在思考"效果。

---

### 5.2 `agent/context.ts` — ContextBuilder

负责组装每次 LLM 调用的完整消息列表。

**System Prompt 构成：**

```
身份声明（平台/运行时/Workspace 路径）
    +
Bootstrap 文件（按顺序合并）
  ├── AGENTS.md   → Agent 行为规范
  ├── SOUL.md     → 角色人格
  ├── USER.md     → 用户偏好
  └── TOOLS.md    → 工具规则
    +
MEMORY.md 长期记忆内容
```

**运行时上下文注入：** 每条用户消息前注入当前时间、Channel、Chat ID，但标记为 `[Runtime Context]` 不存入会话历史，避免污染上下文。

---

### 5.3 `services/llm.ts` — LLMService

OpenAI SDK 封装，兼容所有 OpenAI-compatible API。

| 特性 | 说明 |
|------|------|
| 多模型支持 | 通过 `baseUrl` 切换：OpenAI、Moonshot、DeepSeek、智谱、通义等 |
| 工具调用 | 原生支持 Function Calling，自动解析 tool_calls |
| 自动重试 | 失败后最多重试 3 次，指数退避（1s / 2s / 3s）|
| 类型安全 | 完整 TypeScript 类型，编译时校验 |

---

### 5.4 `tools/registry.ts` + `tools/desktop.ts` — 工具系统

**ToolRegistry：** 统一管理工具注册与执行，自动生成 LLM 所需的 function definitions。

**内置桌面工具：**

| 工具 | 功能 | 安全策略 |
|------|------|---------|
| `open_app` | 打开桌面应用 | 白名单（config.json 按名称匹配）|
| `open_url` | 在浏览器打开网址 | 域名白名单 |
| `open_file` | 用默认程序打开文件 | 路径存在性校验 |
| `open_folder` | 在文件管理器打开文件夹 | 路径存在性校验 |
| `take_screenshot` | 截图保存到本地 | 仅用户主动触发 |
| `send_notification` | 发送系统桌面通知 | 无限制 |
| `get_system_info` | 获取系统信息 | 只读，不修改任何状态 |

---

### 5.5 `session/manager.ts` — SessionManager

JSONL 格式会话历史管理，与原 Python 版本格式**完全兼容**。

- 每个会话一个 `.jsonl` 文件，首行为 metadata（key、时间戳、consolidated 位置）
- 内存缓存 + 磁盘持久化，重启后通过持久化 sessionId 自动恢复历史
- `getHistory()` 从 `lastConsolidated` 位置开始取，确保从 user 消息对齐

---

### 5.6 `memory/store.ts` — MemoryStore

管理 `workspace/memory/MEMORY.md` 文件的读写。

- `getMemoryContext()` — 读取全文，注入到 System Prompt
- `write(content)` — 覆盖写（通过 Agent 工具调用触发）
- `append(entry)` — 追加写

---

### 5.7 `heartbeat/service.ts` — HeartbeatService

周期性唤醒 Agent 检查是否有待处理任务，默认**关闭**。

**两阶段执行：**
1. **决策阶段**：读取 `HEARTBEAT.md`，通过 LLM tool call 判断 `skip` / `run`
2. **执行阶段**：仅当决策为 `run` 时，执行完整 Agent 流程，结果通过 IPC 推送到界面

---

### 5.8 `services/config.ts` — ConfigService

**加载优先级：**
```
环境变量 (OPENAI_API_KEY / OPENAI_BASE_URL)
    > userData/config.json
    > 内置默认值
```

**首次运行初始化：**
- 若 `config.json` 不存在，自动生成包含国内 API 示例的配置模板
- 若 workspace 中缺少模板文件，自动从 `workspace-templates/` 复制（不覆盖已有文件）

---

### 5.9 `ipc/handler.ts` — IPC Handler

替代原 WebSocket 桥，直接处理渲染进程的消息。

- 监听 `nanobot:send` → 调用 `AgentLoop.process()`
- 工具调用进度通过 `onProgress` 回调 → 推送 `nanobot:message (partial=true)`
- 最终回复 → 推送 `nanobot:message (partial=false, done=true)`
- 错误 → 推送 `nanobot:message (type='error')`

## 6. IPC 通信协议

渲染进程侧代码（`chat.ts`）**完全不变**，IPC 接口向后兼容。

| 频道 | 方向 | Payload | 说明 |
|------|------|---------|------|
| `nanobot:send` | Renderer → Main | `{ content, id? }` | 发送用户消息 |
| `nanobot:message` | Main → Renderer | `MessagePayload` | Agent 回复（含流式）|
| `nanobot:status` | Main → Renderer | `{ connected: boolean }` | Agent 就绪状态 |
| `nanobot:proc-status` | Main → Renderer | `{ phase, message? }` | 初始化进度 |

**MessagePayload 结构：**

```typescript
{
  type: 'message' | 'error';
  role?: 'assistant';
  sessionId?: string;
  content: string;
  partial: boolean;   // true = 流式中间块，false = 最终回复
  done: boolean;      // 与 partial 互为镜像
}
```

## 10. 快速启动

### 克隆并运行

```bash
git clone <仓库地址>
cd src/desktop-client
npm ci          # 精确复现依赖（使用 package-lock.json）
npm start       # 首次启动自动生成 config.json 和 workspace 模板
```

### 填写 API Key

编辑 `%APPDATA%\desktop-client\config.json`（首次启动后自动生成），在 `agent.apiKey` 填入密钥，重启应用。

### 开发命令

```bash
npm run dev     # 监听模式，自动重编译 + 热重载
npm run build   # 编译 TypeScript → dist/
npm run dist    # 打包为安装程序（.exe / .dmg / .AppImage）
```
